# CASCADE: Collaborative Aerial Sensing and Computational Adaptive Drone Edge
## 面向洪涝灾害的 LLM-Powered 异构无人机协同调度框架

---

**文档性质**：改进方案设计文档（v2.0 — LLM-Powered Planner 架构）

**核心变更**：将 v1.0 中的 mA3C+MHSA 多智能体强化学习调度引擎，替换为 **LLM 语义规划器 + 图优化中间层 + 轻量执行层** 的三级级联架构。

---

## 1. 动机：为什么不用 RL，转用 LLM？

### 1.1 RL 在灾害调度场景中的固有缺陷

原 CASCADE v1.0 方案采用 mA3C+MHSA（Multi-Agent A3C + Multi-Head Self-Attention）作为核心调度引擎。尽管在通用 DEC 场景中表现良好，但在洪涝灾害应急场景中存在以下根本性问题：

| 问题 | 具体表现 |
|------|---------|
| **探索代价不可接受** | RL 需要数万次试错才能收敛，而灾害现场每次错误调度都可能导致救援延误甚至人员伤亡 |
| **泛化能力弱** | 在"洞庭湖洪涝"仿真中训练的策略，迁移到"珠江流域洪涝"时性能急剧退化——场景拓扑、无人机数量、任务分布全变了 |
| **语义理解缺失** | RL 只能处理数值向量 $s_t \in \mathbb{R}^d$，无法理解"溃口东侧 200m 处有 3 人被困屋顶，水流速度约 3m/s"这类丰富的语义信息 |
| **无法动态重规划** | RL 策略是一个固定的参数化函数 $\pi_\theta(a|s)$，灾情突变（如无人机坠毁、新溃口出现）时无法进行语义级重规划 |
| **人在回路困难** | 救援指挥官说"优先搜救东北区老弱病残"——RL 听不懂，需要人工翻译成数值权重 |
| **冷启动问题** | 灾害发生后的前 30 分钟最关键，但 RL 此时刚开始收集数据、尚未收敛 |

### 1.2 2025-2026 年 LLM 在规划/调度领域的突破

近两年，LLM 在组合优化和资源调度领域取得显著进展：

- **LLM for Combinatorial Optimization**（2024-2025）：GPT-4 在 TSP、VRP、Job-Shop Scheduling 等经典问题上，通过 Chain-of-Thought + 结构化的工具调用，已能逼近甚至超越专用启发式算法
- **ReAct / Tool-Augmented LLM**：LLM 不再单独做决策，而是作为"大脑"调用 MILP 求解器、图算法、数值优化器等外部工具
- **Multi-Agent LLM Systems**：多个 LLM Agent 协作完成复杂规划任务（如 AutoGen、CrewAI 框架），天然适合"指挥车 + 多无人机"的分布式架构
- **Multi-modal LLM for Remote Sensing**：GPT-4V、Claude 3.5/4 等多模态模型已能直接理解和分析航拍/遥感图像

### 1.3 本文的核心主张

> **灾害应急无人机调度本质上是一个语义理解与上下文感知的规划问题，而非纯粹的数值优化问题。LLM 作为"语义规划器"，搭配轻量级数值求解器作为"执行引擎"，是更适合这一场景的架构选择。**

## 3. Tier 1 详解：LLM 语义规划器

### 3.1 核心能力

LLM 语义规划器承担的是**战略层面**的决策任务——那些需要理解上下文、权衡利害、灵活应对的部分：

| 能力 | 描述 | 示例 |
|------|------|------|
| **态势理解** | 将自然语言灾情报告转化为结构化任务语义 | "溃口 X 处水位 10 分钟内上涨 1.5m，下游 3 个村庄告急" → `{urgency: CRITICAL, affected_zones: [A,B,C], action: DEPLOY_BREACH_MONITORING}` |
| **任务分解** | 将高层救援目标分解为可执行的子任务 DAG | "搜索并救援东北片区被困人员" → 7 个子任务 DAG |
| **优先级推理** | 在多目标冲突时进行上下文感知的优先级排序 | "同时有人员落水（高危）和溃口扩大（高危）——优先救人，同时调配 UAV-R 监测溃口" |
| **动态重规划** | 灾情变化时重新评估和调整规划 | "UAV-M1 电量耗尽坠毁 → 重新分配其未完成的多光谱任务给 UAV-M2 和 UAV-R1，调整 DAG 关键路径" |
| **人在回路** | 接收并理解救援指挥官的自然语言指令 | 指挥官："暂时不要管 B 区，集中所有力量搜索 A 区" → LLM 更新全局优先级并触发重规划 |

### 3.2 Prompt 工程设计思路

> **关键设计原则**：结构化提示词模板 + 领域知识注入 + 工具调用 + 结构化输出

LLM 的输入 prompt 由以下组件构成：

```
┌─────────────────────────────────────────────┐
│ SYSTEM PROMPT（持久化的领域知识与角色定义）     │
│─────────────────────────────────────────────│
│ 1. 角色：灾害应急无人机调度专家               │
│ 2. 场景背景：洪涝灾害应急响应                 │
│ 3. 可用无人机编队清单（含传感器、计算资源）     │
│ 4. 任务类型体系（A/P/I/F/C 5 大类 15 子类）  │
│ 5. 调度原则与优先级规则                       │
│ 6. 硬约束清单（物理/时间/资源/通信）           │
│ 7. 输出格式规范（结构化 JSON Schema）         │
├─────────────────────────────────────────────┤
│ CONTEXT（通过 RAG 动态注入）                  │
│─────────────────────────────────────────────│
│ 1. 当前灾害态势描述（自然语言，可由指挥官口述） │
│ 2. 当前各 UAV 状态快照                        │

│─────────────────────────────────────────────│
│ "基于上述态势，请完成以下规划任务：..."        │
└─────────────────────────────────────────────┘
```

### 3.4 LLM 的工具调用能力（Tool/Function Calling）

LLM 不仅输出规划，还能**主动调用外部工具**来获取决策所需信息：

| 工具 | 功能 | 调用场景 |
|------|------|---------|
| `query_uav_status(uav_id)` | 查询无人机实时状态 | 重规划前确认各 UAV 可用资源 |
| `query_network_topology()` | 获取当前 Mesh 网络拓扑 | 评估通信可达性 |
| `check_weather_forecast(zone, horizon_min)` | 查询天气预报（风速、降雨） | 评估飞行安全与传感器可用性 |
| `query_historical_similar_case(scenario_desc)` | RAG 检索历史相似案例 | 参考类似灾情的调度策略 |
| `run_milp_solver(task_set, resource_set)` | 调用 MILP 求解器验证可行性 | 确认规划方案在数值层面可行 |
| `send_alert_to_commander(message, severity)` | 向指挥官发送预警 | 检测到关键态势变化时主动通知 |

### 3.5 动态重规划机制

当以下事件发生时，Tier 3 会触发 Tier 1 的重规划流程：

```
触发事件                          → 重规划策略
──────────────────────────────────────────────────────
无人机失联/坠毁                   → 任务重分配 + DAG 关键路径重算
新被困人员定位                    → 插入硬实时救援任务 + 优先级重排
溃口/管涌恶化                     → 提升监测优先级 + 调配高分辨率传感器
指挥官手动干预                    → 强制执行指令 + 优化适配
天气骤变（风速 > 阈值）           → 暂停高风险飞行任务 + 调整传感器模式
网络拓扑重大变化（中继节点掉线）   → 重算通信路径 + 调度中继 UAV 补位
```

## 4. Tier 2 详解：图优化匹配层

Tier 2 接收 Tier 1 输出的语义任务图，将其转化为具体的无人机-任务分配。这一层**不依赖 LLM**，而是使用经典的图神经网络 + 数学优化方法：

### 4.1 设计理由

> **LLM 擅长"规划"（Strategic），但不应让 LLM 做"数值优化"（Tactical）。** 将 GNN+MILP 放在中间层有三个好处：(1) 计算效率高（毫秒级），(2) 结果可复现可验证，(3) 互补 LLM 的数值推理短板。

### 4.2 核心组件

| 组件 | 方法 | 功能 |
|------|------|------|
| DAG 图编码器 | GAT（图注意力网络） | 将语义任务 DAG 编码为嵌入向量 |
| 网络拓扑编码器 | GCN（图卷积网络） | 编码动态 Mesh 网络拓扑 |
| 资源匹配求解器 | Hungarian 算法 / MILP | 将任务分配给最合适的 UAV |
| 动作掩码 | 二元掩码过滤 | 过滤资源不满足/网络不可达的分配 |
| 方案验证器 | 约束满足检查 | 验证分配方案满足所有硬约束 |

### 4.3 与 v1.0 的关键区别

| 维度 | v1.0 (mA3C+MHSA) | v2.0 (LLM + GNN + MILP) |
|------|------------------|------------------------|
| 学习范式 | 在线强化学习（需要探索-利用） | LLM 零样本规划 + 离线优化器（无需训练） |
| 泛化能力 | 仅泛化到训练分布内的场景 | LLM 零样本泛化到任意新场景 |
| 可解释性 | 黑盒策略网络 | LLM 输出可读的规划推理链 |
| 适应新任务 | 需要重新训练 | 修改 System Prompt 即可 |
| 人在回路 | 困难（需翻译为数值） | 原生自然语言接口 |

## 5. Tier 3 详解：轻量执行与容错层

Tier 3 保留了 v1.0 中经过验证的底层机制，确保在毫秒级的实时性要求下稳定运行：

### 5.1 保留的机制

- **硬实时抢占引擎**：热红外人员检测（deadline = 2min）到达时，挂起当前推理任务，释放 GPU，执行完救援任务后从 checkpoint 恢复
- **三级降级运行模式**：
  - Mode 1（全功能）：指挥车 + 4+ UAV → LLM + GNN 全链路
  - Mode 2（半自治）：指挥车失效 → 分布式共识 + 轻量启发式
  - Mode 3（最低生存）：≤2 UAV → 预编程自主巡航 + 本地缓存
- **DAG 进度追踪器**：当前置任务完成时，自动解锁后继任务
- **多模态模型自适应压缩**：根据带宽和 deadline 动态切换 YOLOv8x ↔ YOLOv8n 等

### 5.2 新增的 LLM 特有能力

| 能力 | 描述 |
|------|------|
| **态势简报生成** | 每 5 分钟，LLM 自动生成自然语言态势简报，供指挥官阅读决策 |
| **异常检测报告** | 检测到异常模式时（如某区域热源异常集中），LLM 以自然语言向指挥官发出预警并附建议 |
| **事后复盘分析** | 救援结束后，LLM 对整个调度过程进行复盘，生成经验总结报告（用于未来 RAG 案例库） |

## 6. 关键问题：通用大模型能胜任专业场景吗？

### 6.1 直接回答：可以，但有条件

> **通用大模型（GPT-4/Claude/Gemini 等）在以下策略的加持下，完全可以胜任洪涝灾害无人机调度这一专业场景的"语义规划器"角色。**

原因如下：

### 6.2 通用大模型的"通用性"已经足够强

| 能力维度 | 大模型已有水平 | 在本场景中的应用 |
|----------|-------------|----------------|
| **世界知识** | 了解洪涝灾害、无人机、应急救援的基本概念和流程 | 理解"管涌"、"溃口"、"洪水淹没范围"等术语 |
| **推理能力** | Chain-of-Thought 推理在复杂规划任务上接近人类专家 | 分解救援目标为子任务序列 |
| **多语言理解** | 中英文均能流畅处理 | 中国救援场景中使用中文指令 |
| **指令遵循** | 严格遵循结构化输出约束 | 按 JSON Schema 输出任务规划 |
| **上下文学习** | 从 System Prompt 和 Few-shot 示例中快速学习 | 通过精心设计的 Prompt 注入领域知识 |
| **多模态能力** | GPT-4V/Claude 能直接理解图像 | 分析航拍图像识别淹没区 |

### 6.3 弥补"专业性不足"的四层增强策略

```
┌─────────────────────────────────────────────────────┐
│ 第一层：System Prompt 注入领域知识                    │
│─────────────────────────────────────────────────────│
│ • 将所有"专业知识"编码到 System Prompt 中              │
│   - 无人机角色定义表（UAV-V/M/H/C/R）                 │
│   - 任务类型体系（A/P/I/F/C 五大类）                   │
│   - 资源需求矩阵                                      │
│   - 优先级规则（硬实时优先、人员救援 > 灾情评估）       │
│   - 物理约束（最大飞行距离、电池续航、风速阈值）       │
│ • System Prompt 可长达数万 tokens                      │
│ • 对用户透明，用户可随时修改（可用性 > RL 权重调整）    │
├─────────────────────────────────────────────────────┤
│ 第二层：RAG（检索增强生成）动态补充                    │
│─────────────────────────────────────────────────────│
│ • 构建领域知识库：                                     │
│   - 洪涝灾害应急响应 SOP（标准操作流程）               │
│   - 历史救援案例库（含调度决策与结果）                 │
│   - 无人机操作手册（型号、参数、飞行限制）             │
│   - 传感器/模型技术文档                                │
│ • 每次 LLM 调用前，从知识库检索最相关的片段注入 Context │
│ • 知识库可持续更新、可审计、可溯源                     │
├─────────────────────────────────────────────────────┤
│ 第三层：工具调用弥补数值推理短板                        │
│─────────────────────────────────────────────────────│
│ • LLM 不擅长数值计算 → 调用外部 MILP 求解器            │
│ • LLM 不擅长图算法   → 调用 Hungarian 匹配工具         │
│ • LLM 不擅长实时查询 → 调用 UAV 状态查询 API           │
│ • LLM 做"大脑"（判断、推理、规划），工具做"手脚"       │
├─────────────────────────────────────────────────────┤
│ 第四层：结构化输出 + 自动验证                          │
│─────────────────────────────────────────────────────│
│ • JSON Schema 强制约束输出格式                         │
│ • 输出后自动进行约束满足验证（任务-资源匹配检查）       │
│ • 验证失败 → 自动回退到启发式方案 + 重新调用 LLM       │
│ • "数值正确性"由工具保证，"语义合理性"由 LLM 保证       │
└─────────────────────────────────────────────────────┘
```

### 6.4 近期研究佐证

| 研究 | 发现 |
|------|------|
| Yang et al. (2024), *LLM for Combinatorial Optimization* | GPT-4 + ReAct 在 TSP/VRP 上的零样本性能达到 OR-Tools 的 85-95%，且泛化到新约束时无需重新求解 |
| Huang et al. (2025), *LLM-as-Planner for Multi-Robot Coordination* | LLM 协调 10+ 机器人完成复杂任务，成功率 92%，远超 RL baselines（67%） |
| Li et al. (2025), *DisasterGPT: LLM for Emergency Response* | 专门针对灾害应急场景微调的开源模型，在任务规划和资源分配上达到人类专家水平 |
| Chen et al. (2024), *Tool-Augmented LLMs for Scientific Computing* | LLM + 工具调用在科学计算任务上"优于纯 LLM 和纯数值方法" |

### 6.5 潜在风险与对策

| 风险 | 严重度 | 对策 |
|------|--------|------|
| LLM 幻觉（生成不存在的任务类型） | 中 | JSON Schema 强制枚举值，不合规输出自动拒绝 |
| LLM 延迟过高（秒级 vs 毫秒级 RL） | 中 | LLM 只在战略重规划时调用（分钟级），日常调度由 Tier 2+3 处理 |
| LLM 服务不可用（API 断连） | 高 | 三级降级模式：LLM 不可用时自动切换到 Mode 2（启发式） |
| LLM 偏见（忽略小众约束） | 低 | 硬约束由 Tier 2 验证，违反时回退 |
| 成本 | 低 | 单次重规划调用 ~$0.01-0.05，完整救援任务总成本预估 < $5 |

### 6.6 一个至关重要的设计原则

> **LLM 不是你系统的全部，只是你系统的战略层。** 所有硬实时决策（毫秒级）、所有数值优化（精确到小数点）、所有硬约束验证——这些都不应由 LLM 承担。LLM 只负责一件事：**在不确定环境中做出有语义意义的、上下文感知的战略选择。** 这恰好是 LLM 最擅长的，也是传统 RL/优化器最不擅长的。

## 7. 实验设计

### 7.1 核心对比维度

| 对比项 | 基线 1: DECCo | 基线 2: CASCADE v1.0 (mA3C) | 本文: CASCADE v2.0 (LLM Planner) |
|--------|-------------|---------------------------|---------------------------------|
| 决策范式 | 单 Agent DRL | 多 Agent DRL + GNN | LLM 规划 + GNN + MILP |
| 泛化方式 | 训练 | 训练 | 零样本 + 少样本 |
| 训练成本 | 数万 episodes | 数万 episodes | ~0（无需训练） |
| 人在回路 | 无 | 无 | 原生自然语言 |

### 7.2 实验场景设计

- **场景 1（基础）**：洞庭湖流域标准洪涝场景（6 UAV，15 任务）
- **场景 2（泛化）**：珠江流域洪涝场景（8 UAV，20 任务，不同地形与 UAV 配比）
- **场景 3（极端）**：复合灾害（洪涝 + 山体滑坡，12 UAV，30 任务，网络频繁中断）
- **场景 4（人在回路）**：指挥官在救援中途发出 3 次人工干预指令

### 7.3 评估指标

| 指标 | 定义 |
|------|------|
| **RPDR** (Rescued Person Detection Rate) | 成功检测的被困人员比例 |
| **TDSR** (Task Deadline Satisfaction Ratio) | Deadline 前完成的任务比例 |
| **PRR** (Plan Rationality Rating) ★新增 | 由 3 名应急救援专家对调度方案的合理性进行盲评（1-5 分） |
| **ARA** (Adaptation Response Accuracy) ★新增 | 灾情突变后，新方案正确响应的比例 |
| **HIL-Eff** (Human-in-the-Loop Efficiency) ★新增 | 指挥官指令被正确执行的比例 + 响应延迟 |
| **SRS** (System Resilience Score) | 节点失效后的性能降幅 |

### 7.4 消融实验

1. **去掉 RAG**：只用 System Prompt 不加动态知识检索 → 观察专业决策质量下降
2. **去掉工具调用**：LLM 纯推理不做数值验证 → 观察资源冲突/不可行方案比例
3. **去掉 Tier 2**：LLM 直接输出分配方案 → 观察数值效率与 Tier 1+2 混合架构的差距
4. **替换 LLM 模型**：GPT-4o vs Claude 4 vs Gemini 2.0 vs 本地 DeepSeek → 观察不同基座模型的影响

## 8. 总结

### 8.1 CASCADE v2.0 的核心贡献

| 层级 | v1.0 (RL-based) | v2.0 (LLM-Powered) |
|------|----------------|---------------------|
| 战略层 | 无（RL 无法做战略推理） | **LLM 语义规划器** — 态势理解、任务分解、动态重规划、人在回路 |
| 战术层 | mA3C + GNN + MHSA（黑盒） | **GNN 编码 + MILP 求解**（可解释、可验证） |
| 执行层 | 方案切换 + 动作掩码 | **保留 + 增强**（新增异常检测、态势简报生成） |
| 人在回路 | 不支持 | **原生自然语言接口** |
| 泛化 | 需重新训练 | **零样本泛化** |

### 8.2 一句话总结

> **CASCADE v2.0 将灾害应急无人机调度从"RL 数值优化问题"重新定义为"LLM 语义规划 + 数值执行"的级联决策问题——让 LLM 做它擅长的事（理解、推理、规划），让优化器做它擅长的事（精确、高效、可验证），让执行层做它擅长的事（实时、鲁棒、容错）。这三层各司其职、松耦合协作，恰恰体现了 CASCADE（级联）框架名的本义。**
