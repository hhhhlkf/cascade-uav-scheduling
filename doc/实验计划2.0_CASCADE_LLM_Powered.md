## 1. 实验目标

### 1.1 核心验证目标

| 编号 | 目标 | 对应方案文档章节 |
|------|------|-----------------|
| **G1** | CASCADE v2.0 (LLM-Powered) 在救援效能 (RPDR) 上显著优于 CASCADE v1.0 (mA3C+MHSA) 及其他 DRL 方法，无需任何训练 | 方案 1.1, 4.3 |
| **G2** | LLM 语义规划器在灾情突变（UAV 坠毁、新紧急任务插入）时具备零样本动态重规划能力，泛化到训练中未见过的新场景 | 方案 3.1, 3.5 |
| **G3** | 四层增强策略（System Prompt + RAG + 工具调用 + 结构化输出）各自对 LLM 规划质量有统计显著的贡献 | 方案 6.3 |
| **G4** | Tier 2 (GNN+MILP) 对方案可执行性和数值效率有显著贡献——去掉 Tier 2 后资源冲突率显著上升 | 方案 4.1, 4.2 |
| **G5** | 人在回路 (HIL) 接口使指挥官自然语言指令的正确执行率达到 90%+ | 方案 3.1, 3.5 |
| **G6** | 不同基座 LLM（GPT-4o, Claude 4, Gemini 2.0, DeepSeek-V3）在规划质量上存在差异，但均可通过增强策略达到可用水平 | 方案 6.2 |

### 1.2 实验范围限定

- **全部在仿真环境中进行**，不涉及真实无人机硬件
- 调度决策由 **LLM（Tier 1 战略规划）+ GNN/MILP（Tier 2 战术匹配）+ 规则引擎（Tier 3 执行容错）** 三级协作完成
- LLM 调用通过 API（OpenAI / Anthropic / Google / DeepSeek），成本需记录
- 仿真环境与 v1.0 实验计划**共享**（UAV 模拟器、通信网络、任务管理器），仅调度算法层不同

class CASCADEEnvV2(gym.Env):
    """
    与 v1.0 共享底层仿真引擎，差异在于:
      - 动作空间接受 ScheduleAction（含结构化分配方案），而非原始数值向量

            "sim_time": self.sim_time,
            "situation": {
                "disaster_type": self.scenario.disaster_type,
                "affected_zones": self._summarize_zones(),
                "recent_events": self.event_log[-10:],
                "commander_instructions": self.pending_commander_instructions
            },
            "uavs": [
                {
                    "id": uav.id, "type": uav.type,
                    "position": uav.position,
                    "battery": uav.battery_level,
                    "status": uav.status.value,
                    "current_tasks": [t.id for t in uav.current_tasks],
                    "resources": {
                        "cpu": uav.cpu_available,
                        "gpu": uav.gpu_available,

### 3.2 CASCADE v2.0 (LLM-Powered) — 本文方法

#### 3.2.1 三级架构

```
┌──────────────────────────────────────────────────────────────┐
│ Tier 1: LLM 语义规划器 (调用频率: 分钟级)                      │
│──────────────────────────────────────────────────────────────│
│ 输入: 自然语言灾害态势 + 当前任务状态 + 历史反馈               │
│ 处理: LLM (GPT-4o/Claude/Gemini) + System Prompt + RAG        │
│       + Function Calling (调用仿真环境工具获取实时数据)         │
│ 输出: 语义任务图 G_sem (任务分解, 优先级建议, 约束条件)         │
│ 特点: 零样本、可解释、人在回路                                  │
├──────────────────────────────────────────────────────────────┤
│ Tier 2: GNN + MILP 图优化匹配层 (调用频率: 秒级)               │
│──────────────────────────────────────────────────────────────│
│ 输入: 语义任务图 G_sem + 实时资源向量 + 网络拓扑               │
│ 处理: GAT 编码 DAG + GCN 编码拓扑 + Hungarian/MILP 匹配        │
│ 输出: 具体 UAV-任务分配方案 + 执行时序                          │
│ 特点: 精确、高效、可验证                                        │
├──────────────────────────────────────────────────────────────┤
│ Tier 3: 轻量执行与容错层 (调用频率: 毫秒级)                     │
│──────────────────────────────────────────────────────────────│
│ 输入: 分配方案 + 实时传感器/网络状态                            │
│ 处理: 硬实时抢占 + 模型自适应压缩 + 三级降级 + DAG 进度追踪     │
│ 输出: 执行指令 + 异常处理 + 态势简报 (LLM 生成)                 │
│ 特点: 实时、鲁棒、容错                                          │
└──────────────────────────────────────────────────────────────┘
```

#### 3.2.2 CASCADE LLM Scheduler 核心实现

```python
class CASCADELlmscheduler(BaseScheduler):
    """CASCADE v2.0: LLM-Powered 三级级联调度器"""

    def __init__(self, config):
        # === LLM 客户端（通过 LiteLLM 统一多模型）===
        self.llm = LiteLLMClient(
            model=config.llm.model,          # gpt-4o / claude-4 / gemini-2.0
            temperature=config.llm.temperature,
            max_tokens=config.llm.max_tokens
        )

        # === Prompt 构建器 ===
        self.prompt_builder = PromptBuilder(
            system_prompt_path=config.prompt.system_path,
            schema_path=config.prompt.output_schema_path
        )

        # === RAG 检索器 ===
        self.rag = RAGRetriever(
            embedding_model=config.rag.embedding_model,
            knowledge_base_path=config.rag.kb_path,
            top_k=config.rag.top_k
        )

        # === Tier 2: 数值优化组件 ===
        self.gnn_encoder = GNNEncoder()       # GAT + GCN
        self.hungarian = HungarianMatcher()
        self.milp = MILPSolver(time_limit_ms=500)
        self.constraint_checker = ConstraintChecker()

        # === Tier 3: 执行容错组件 ===
        self.preemption_engine = PreemptionEngine()
        self.degradation_manager = DegradationManager()
        self.model_compressor = ModelCompressor()

        # === 规划缓存 ===
        self.current_plan: Optional[SemanticTaskGraph] = None
        self.plan_timestamp: float = 0.0
        self.min_replan_interval: float = 30.0  # 秒
        self.replan_triggers = []                # 重规划触发事件列表

    def decide(self, env: CASCADEEnvV2) -> ScheduleAction:
        # === Tier 3 检查: 硬实时抢占（毫秒级，不经过 LLM）===
        preempt = self.preemption_engine.check(env)
        if preempt:
            return preempt

        # === 判断是否需要 LLM 重规划 ===
        need_replan = (
            self.current_plan is None
            or self._emergency_detected(env)
            or (env.sim_time - self.plan_timestamp) > self.min_replan_interval
            or self._commander_override_pending(env)
        )

        if need_replan:
            # ===== Tier 1: LLM 语义规划 =====
            context = env.get_llm_context()
            rag_docs = self.rag.retrieve(context["situation"])
            tools = env.get_available_tools()

            llm_response = self.llm.chat_with_tools(
                system_prompt=self.prompt_builder.build_system(context),
                user_prompt=self.prompt_builder.build_user(context, rag_docs),
                tools=tools,
                response_schema=SemanticTaskGraph.schema()
            )
            self.current_plan = self._parse_response(llm_response)
            self.plan_timestamp = env.sim_time

        # ===== Tier 2: GNN + MILP 最优匹配 =====
        task_emb = self.gnn_encoder.encode_tasks(self.current_plan)
        uav_emb = self.gnn_encoder.encode_uavs(env.uav_states)
        net_emb = self.gnn_encoder.encode_network(env.network)

        assignments = self.hungarian.match(
            tasks=env.get_ready_tasks(),
            uavs=env.uav_states,
            task_emb=task_emb,
            uav_emb=uav_emb,
            net_emb=net_emb,
            plan_constraints=self.current_plan.constraints
        )

        if not self.constraint_checker.verify(assignments):
            assignments = self.milp.solve(
                env.get_ready_tasks(), env.uav_states
            )

        return ScheduleAction(
            assignments=assignments,
            strategy=self.current_plan.strategy,
            plan_id=self.current_plan.plan_id
        )

    def handle_commander_instruction(self, instruction: str) -> Dict:
        """处理指挥官自然语言指令"""
        # 直接注入 LLM，触发即时重规划
        self.current_plan = None  # 强制下个 decide() 重规划
        return {"status": "acknowledged", "instruction": instruction}
```

#### 3.2.3 LLM Prompt 结构

```
┌──────────────────────────────────────────────┐
│ SYSTEM PROMPT (持久化 ~8K tokens)              │
│──────────────────────────────────────────────│
│ 1. 角色定义：洪涝灾害无人机应急调度专家         │
│ 2. 领域知识（完整注入，减少 RAG 负担）：         │
│    - 无人机角色定义表 (UAV-V/M/H/C/R)          │
│    - 任务类型体系 (A/P/I/F/C 5 大类 15 子类)   │
│    - 资源需求矩阵                               │
│    - 调度原则：(1) 人员救援 > 灾情评估          │
│               (2) 硬实时 (I2) > 软实时         │
│               (3) 能量效率 ≈ 时效性             │
│    - 硬约束清单 (飞行距离/电池/风速/并发数)     │
│ 3. 输出格式规范 (JSON Schema，严格校验)         │
│ 4. 工具使用说明 (何时调用哪个 Function)         │
├──────────────────────────────────────────────┤
│ USER PROMPT (每次调用动态构建 ~2K tokens)       │
│──────────────────────────────────────────────│
│ 1. 当前态势摘要                               │
│ 2. RAG 检索到的相关案例/SOP 片段               │
│ 3. 最新事件日志 (最近 10 条)                   │
│ 4. 指挥官指令 (如有)                           │
│ 5. 指令："基于以上信息，请输出结构化任务规划"   │
└──────────────────────────────────────────────┘
```

## 4. 对比方法

### 4.1 方法总览

| 类别 | 方法 | 说明 | 优先级 |
|------|------|------|--------|
| **本文方法** | **CASCADE v2.0 (LLM-Powered)** | LLM T1 + GNN/MILP T2 + 规则 T3 | P0 |
| 本文变体 | CASCADE v1.0 (mA3C+MHSA) | 纯 RL，来自 v1.0 实验计划 | P0 |
| DRL | DECCo (mA2C) | Zhang et al. 2023 | P0 |
| DRL | MADDPG | Lowe et al. 2017 | P0 |
| DRL | QMIX | Rashid et al. 2018 | P1 |
| 启发式 | Greedy / GA / Min-Load / Round-Robin / HEFT | 传统方法 | P0 |
| 优化 | MILP (OR-Tools) | 数学优化上界 | P1 |
| LLM 消融 | **LLM-Direct** | 去掉 Tier 2，LLM 直接输出分配 | P0 |
| LLM 消融 | **LLM-NoRAG** | 去掉 RAG，仅 System Prompt | P0 |
| LLM 消融 | **LLM-NoTools** | 去掉 Function Calling | P0 |
| LLM 消融 | **LLM-ZeroShot** | 去掉 System Prompt 领域知识注入 | P1 |
| LLM 模型 | GPT-4o / Claude 4 / Gemini 2.0 / DeepSeek-V3 | 不同基座模型对比 | P0 |

### 4.2 核心对比维度

| 维度 | DECCo | CASCADE v1.0 (RL) | CASCADE v2.0 (LLM) |
|------|-------|-------------------|---------------------|
| 决策范式 | 单 Agent DRL | 多 Agent DRL + GNN | LLM 规划 + GNN + MILP |
| 训练需求 | 数万 episodes | 数万 episodes | **0 episodes（零样本）** |
| 泛化方式 | 分布内 | 分布内 | **零样本跨场景泛化** |
| 可解释性 | 黑盒 | 黑盒 | **自然语言推理链** |
| 人在回路 | 不支持 | 不支持 | **原生自然语言接口** |
| 适应新任务 | 需重新训练 | 需重新训练 | **修改 Prompt 即可** |

## 5. 实验设计

### 5.1 实验场景

与 v1.0 共享 4 个场景，额外增加 **自然语言描述变体**（S1-NL, S2-NL）：

|------|---------|-------------|
| **S1 (ID)** | 洞庭湖标准洪涝 (5×5km, 10 UAV, 30 任务) | S1-NL: 指挥官口述版（"洞庭湖那边堤坝决口了，A片区水位涨得很快..."） |

### 5.3 RL 方法的训练配置

RL baseline（CASCADE v1.0, DECCo, MADDPG, QMIX）的训练配置同 v1.0 实验计划。LLM 方法**不需要训练**，直接零样本评估。

### 5.4 LLM 调用配置

| 参数 | 值 | 说明 |
|------|-----|------|
| LLM 模型 | GPT-4o (默认) / Claude 4 / Gemini 2.0 / DeepSeek-V3 | E8 中切换 |
| Temperature | 0.3 | 低温度保证输出稳定性 |
| Max tokens | 4096 | 足够输出完整 JSON 规划 |
| System Prompt 长度 | ~8K tokens | 领域知识全量注入 |
| User Prompt 长度 | ~2K tokens | 动态态势 + RAG 片段 |
| RAG top-k | 3 | 检索最相关的 3 个文档片段 |
| Function Calling 最大轮次 | 5 | LLM 可进行多轮工具调用 |
| 单次 LLM 调用超时 | 30s | 超时则使用缓存的上次规划 |
| LLM 不可用降级 | Mode 2 (启发式) | 方案 5.2/6.5 的对策 |

### 5.5 每次实验的运行流程

```
1. 加载实验配置
2. for method in methods:
3.     for seed in seeds (5 seeds):
4.         env = CASCADEEnvV2(scenario_config, seed)
5.         scheduler = Method(config)
6.
7.         if method is RL-based:
8.             # 需要训练（同 v1.0 流程）
9.             train(env, scheduler, N_episodes)
10.            eval_metrics = evaluate(env_test, scheduler, N_eval_episodes)
11.        elif method is LLM-based:
12.            # 零样本评估，无需训练
13.            eval_metrics = evaluate(env_test, scheduler, N_eval_episodes)
14.            # 额外记录 LLM 调用统计
15.            eval_metrics["llm_calls"] = scheduler.stats.total_llm_calls
16.            eval_metrics["llm_cost_usd"] = scheduler.stats.total_cost_usd
17.            eval_metrics["llm_avg_latency_s"] = scheduler.stats.avg_latency_s
18.            eval_metrics["format_error_rate"] = scheduler.stats.format_errors / scheduler.stats.total_calls
19.        else:
20.            # 启发式/优化方法，直接评估
21.            eval_metrics = evaluate(env_test, scheduler, N_eval_episodes)
22.
23.        mlflow.log_metrics(eval_metrics)
24.
25.    # 汇总 5 seed
26.    summary = aggregate(test_results)
```

### 6.1 救援效能指标（与 v1.0 共享）

| 指标 | 定义 | 目标 |
|------|------|------|
| **RPDR** | 成功检测的被困人员比例 | ↑ |

| **ART** | 救援类任务的平均响应时间 | ↓ |
| **SRS** | 节点失效后 RPDR 下降百分比 | ↓ |
| **Energy Efficiency** | 单位能量完成的有效任务数 | ↑ |

### 6.2 LLM 专有指标（v2.0 新增）

| 指标 | 定义 | 目标 |
|------|------|------|
| **PRR** (Plan Rationality Rating) | 3 名应急救援专家对调度方案的盲评 (1-5) | ↑ |
| **ARA** (Adaptation Response Accuracy) | 灾情突变后新方案正确响应的比例 | ↑ 最大化 |
| **HIL-Eff** (Human-in-the-Loop Efficiency) | 指挥官指令被正确执行的比例 | ↑ ≥ 90% |
| **FER** (Format Error Rate) | LLM 输出 JSON 格式错误的比例 | ↓ |
| **RCR** (Resource Conflict Rate) | 调度方案中出现资源冲突的 step 比例 | ↓ |
| **LLM Latency** | 单次 LLM 调用平均延迟 (s) | 记录 |
| **LLM Cost** | 单次实验 LLM API 总费用 ($) | 记录 |
| **Replan Frequency** | 每小时触发 LLM 重规划的次数 | 记录 |

### 6.3 与 v1.0 RL 方法的特有对比维度

| 指标 | v1.0 (RL) | v2.0 (LLM) | 期望 |
|------|-----------|------------|------|
| 训练时间 | 数小时 | **0** | v2.0 >> v1.0 |
| 泛化到 S2（不重训） | RPDR 大幅下降 | **RPDR 基本持平** | v2.0 >> v1.0 |
| 跨场景可解释性 | 无 | **自然语言推理链** | v2.0 独有 |
| 对新任务类型的适应性 | 需修改网络重训 | **修改 Prompt 即可** | v2.0 >> v1.0 |
| API 成本 | $0 | < $5/experiment | 可接受 |
