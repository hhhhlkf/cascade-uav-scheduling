# DECCo 论文分析与面向中国洪涝灾害场景的多模态多任务调度改进方案（CASCADE）

---

## 一、DECCo 论文核心内容总结

### 1.1 研究背景与问题

Zhang 等人（2023）发表于 *Drones* 期刊的 DECCo（**D**rone **E**dge **C**luster **Co**llaborative Task Scheduling Framework），面向**异构无人机边缘集群（Drone Edge Cluster, DEC）** 场景，解决动态变化任务请求下的协同任务调度问题。

传统 Kubernetes 等分布式边缘计算框架在 DEC 场景中有两个主要不足：
- 无人机节点在硬件资源（CPU、内存）和网络资源（通信延迟、带宽）上存在强异构性；
- 现有框架通常只针对单一指标调度，无法综合描述 DEC 全局状态与无人机局部状态。

### 1.2 方法设计

**框架架构**（三组件）：

| 组件 | 功能 |
|------|------|
| **Cluster Orchestrator**（集群编排器） | 节点管理、全局状态监控、任务接收与日志记录 |
| **Co-Scheduler**（协同调度器） | 核心——包含 mA2C（DRL 算法）+ 启发式算法（遗传算法）+ 方案切换单元 |
| **Cluster Controller**（集群控制器） | 直接连接无人机，执行具体调度操作与服务部署 |

**MDP 建模**：

- **状态空间** $s_t$：当前任务的 CPU/内存需求 $E_t$、任务队列 $Q_t$、各无人机可用资源 $A_{u,t}$ 和硬件资源 $H_u$、通信延迟 $D_t$
- **动作空间**：选择第 $i$ 架无人机或云端执行当前任务
- **成本函数** $C = \lambda_1 C_1 + \lambda_2 C_2 + \lambda_3 C_3$：
  - $C_1$：负载均衡成本（CPU + 内存资源标准差）
  - $C_2$：资源过载惩罚（带权重系数 $k > 1$）
  - $C_3$：通信延迟成本

**mA2C 算法**（改进的 Advantage Actor-Critic）：

- **Base Station Critic**：使用评估网络 $Q(\cdot,\cdot;\theta_e)$ 和 Target 网络 $Q(\cdot,\cdot;\theta_t)$ 求解状态价值函数，结合经验回放池降低方差
- **UAV Actor**：基于策略梯度方法更新调度策略，引入基线函数 $B(s_t) = v(s_t;\theta_e)$ 计算优势函数
- **动作掩码机制**：通过二元向量 $F_t \in \{0,1\}^{U+1}$ 过滤不可用节点（掉线/超延迟），对策略网络输出做逐元素乘法 $\hat{p}(s_t) = p(s_t) \odot F_t$，避免将任务调度到不可用节点

**方案切换单元（Solution Switching Unit）**：

当 mA2C 连续 100 次迭代优于启发式方案时，自动从启发式方案切换到 mA2C 决策，避免 DRL 探索初期的严重次优调度。

### 1.3 实验评估

使用 Google Cluster Usage Traces（GCT）数据集，在 Google Cloud Platform 上构建 1 个主节点 + 8 个边缘节点的集群。对比实验表明：
- 初期（slot 1~1000）：启发式方案优于 mA2C（DRL 需要探索）
- 中后期（slot > 2260）：mA2C 显著优于启发式方案
- 负载均衡最优参数：$\lambda_1=0.3, \lambda_2=0.5, \lambda_3=0.2$

## 二、DECCo 在应急灾害场景中的不足分析

DECCo 是一个优秀的通用 DEC 任务调度框架，但直接应用于**中国应急灾害无人机遥感场景**时存在以下不适应性：

### 2.1 任务模型过于简单

DECCo 将任务抽象为统一的 CPU/内存需求向量，而灾害应急场景中，无人机承载的任务类型差异极大：

| 任务类型 | 计算特征 | DECCo 能否建模 |
|----------|----------|:---:|
| RGB 高分辨率图像目标检测 | GPU 密集型（YOLO/DETR 推理） | 仅抽象为 CPU/内存 |
| 热红外图像被困人员搜索 | 低延迟关键任务，GPU 中等 | 无法区分优先级 |
| 多光谱/高光谱灾情识别 | GPU + 内存密集型 | 无法区分资源类型 |
| 多光谱图像损毁评估 | 中等计算，大带宽传输 | 未考虑数据传输 |
| 通信中继 | 极低计算，高网络要求 | 未建模 |
| 多模态融合分析 | GPU + 大内存 + 高带宽 | 超出模型能力 |

**DECCo 的"一刀切"资源模型（仅 CPU + 内存）无法刻画多模态 AI 推理任务对 GPU、NPU、带宽的差异化需求。**

### 2.2 缺乏任务优先级与时效性约束

灾害救援中，"5 分钟内定位被困人员"和"1 小时内完成洪涝灾区的淹没范围与灾情评估图"的时效性要求截然不同。DECCo 的成本函数无 deadline 意识，可能导致关键救援任务被延迟。

### 2.3 缺乏任务依赖与流水线建模

多模态灾害分析天然是流水线结构：

```
多光谱采集 → 辐射校正预处理 → 植被/水体指数计算 → 淹没评估
                                                    ↘
RGB 图像采集 → 拼接预处理 → 目标检测 → 多模态融合 → 洪涝灾情综合研判报告
                                                    ↗
热红外采集 → 温度校准预处理 → 异常热源检测 →
```

DECCo 一次只调度一个独立任务，完全忽略了这种**DAG（有向无环图）依赖结构**。

### 2.4 缺乏多模态数据流建模

不同模态相机的数据产出量和传输需求差异巨大：

| 模态 | 单帧数据量 | 采集频率 | 带宽需求 |
|------|-----------|---------|---------|
| RGB 可见光（4K） | ~8 MB | 1 fps | ~64 Mbps |
| 热红外 | ~0.3 MB | 10 fps | ~24 Mbps |
| 低空正射影像块 | ~12 MB | 1 fps | ~96 Mbps |
| 多光谱（5 波段） | ~20 MB | 1 fps | ~160 Mbps |
| 高光谱（200+ 波段） | ~500 MB | 0.1 fps | ~400 Mbps |

DECCo 仅以"通信延迟"一维指标描述网络状态，无法刻画这种多模态、差异化的数据流。

### 2.5 场景假设不匹配

DECCo 假设基站与无人机之间通过 LAN 互联——这在固定基站部署场景中合理。但灾害场景中：
- 基站可能已被摧毁，无人机需自组网（ad-hoc mesh network）
- 多跳中继、间歇性连接是常态
- 云端可能不可达，边缘计算需求更迫切

### 2.6 单任务单时隙的调度粒度

DECCo 的 $a_t = i$（一个时隙只调度一个任务到一架无人机），这对于灾害场景中需要**同时进行多无人机、多任务、多流水线协同**的需求来说粒度过粗。

## 三、改进方案：CASCADE——面向洪涝灾害的协同级联调度框架（Collaborative Aerial Sensing and Computational Adaptive Drone Edge）

### 3.1 整体场景设定

**灾害类型**：中国南方流域性特大洪涝灾害（模拟 2024 年洞庭湖流域决堤及极端强降雨洪涝场景）

**场景要素**：
- 洪涝后通信基础设施大面积损毁，公网不可用
- 部署**异构无人机集群**（数量与类型根据灾情规模、受灾面积和任务需求动态确定，典型配置 6~15 架）
- 无人机搭载不同类型的多模态传感器
- 配有 1 台**应急通信指挥舟/指挥车**作为移动边缘计算中心（替代 DECCo 的固定基站）
- 洪涝地形复杂（淹没区、孤岛、溃口），无人机之间通过 LoRa/WiFi 6 Mesh 自组网通信
- **应急通信指挥车**同时承担全局协调与区域任务分发职责，按区域统筹派发任务链，汇聚各区域的中间结果

#### 3.1.1 区域化任务分发机制

受灾区域通常面积巨大且地理异质性显著（如溃口附近、淹没建成区、高坡孤岛、次生灾害风险区等），单一全局任务模型无法刻画不同区域的差异化救援需求。因此 CASCADE 采用**区域化任务分发机制**：

```
                         ┌─────────────────────────────────────┐
                         │       应急通信指挥车（全局+分发）      │
                         │   - 灾情总览与区域划分                │
                         │   - 全局资源池管理                    │
                         │   - 按区域派发任务链                  │
                         │   - 跨区域调度协调                    │
                         └────────┬───────┬───────┬────────────┘
                                  │       │       │
                    ┌─────────────┘       │       └─────────────┐
                    │                     │                     │
              ┌─────▼─────┐        ┌─────▼─────┐        ┌─────▼─────┐
              │ 区域 A     │        │ 区域 B     │        │ 区域 C     │
              │ (溃口区域) │        │ (淹没城区) │        │ (孤岛区域) │
              │ 任务链 α   │        │ 任务链 β   │        │ 任务链 γ   │
              │ UAV 集群   │        │ UAV 集群   │        │ UAV 集群   │
              │ (协同执行) │        │ (协同执行) │        │ (协同执行) │
              └───────────┘        └───────────┘        └───────────┘
```

**区域划分原则**：
- 指挥车根据灾前卫星影像与灾后首轮侦察结果，将受灾区域划分为若干子区域
- 指挥车根据区域特征为每个子区域生成定制化的任务链并统一派发
- 各区域根据自身特征（淹没深度、建筑物密度、人口分布、通信条件）生成定制化的**任务链**

**任务链（Task Chain）定义**：
- 每个区域的任务链包含 **5~15 个子任务**，覆盖采集、预处理、推理、融合四类，具体数量由区域面积、灾情复杂度和模态需求动态决定
- 任务链内的子任务存在 DAG 依赖关系（如采集→预处理→推理→融合）
- 不同区域的任务链在模态组成、优先级分布、deadline 紧迫度上体现差异化
- 指挥车负责任务链的按时序派发、各区域进度追踪与中间结果汇总

**典型三区域示例**：

| 区域 | 特征 | 任务链子任务数 | 模态重点 |
|------|------|:---:|---------|
| **A: 溃口区域** | 水流湍急、溃口扩张风险，需实时监测 | 14 | 热红外（人员搜索优先）+ RGB（溃口形变监测）+ 高光谱（水质异常） |
| **B: 淹没城区** | 建筑密集、被困人员多、道路中断 | 15 | RGB（YOLO 目标检测）+ 热红外（屋顶被困人员）+ 多模态融合定位 |
| **C: 高坡孤岛** | 地势较高但被洪水围困，物资需求紧迫 | 11 | RGB（道路通行性评估）+ 多光谱（植被/裸露地表分析）+ 通信中继保障 |

> 区域数量和任务链结构由仿真程序根据灾情参数动态生成，详见 3.8.1 节。

### 3.2 异构无人机角色定义

所有无人机均搭载 **NVIDIA Jetson Orin NX** 边缘计算板卡，按显存容量分为两个规格：

| 规格 | 显存 | AI 算力 | 适用场景 |
|------|------|---------|---------|
| **Orin NX 8GB** | 8 GB (102.4 GB/s) | 70 TOPS (sparse) | 轻量推理、采集预处理、通信中继 |
| **Orin NX 16GB** | 16 GB (102.4 GB/s) | 100 TOPS (sparse) | 多模态融合、高光谱分析、大模型推理 |

> Orin NX 是当前无人机边缘计算的主流选择，功耗仅 10~25W，适合机载部署。统一板卡架构简化了硬件异构性管理，使调度器关注的核心矛盾从"不同算力架构的适配"转移到"显存、带宽、任务优先级的合理分配"。

| 无人机类型 | 典型数量 | 搭载传感器 | 计算板卡 | 角色 |
|-----------|------|-----------|-------------|------|
| **UAV-V**（视觉侦察） | 3 | RGB 4K 相机 + 热红外 | Orin NX 16GB | 目标检测、被困人员搜索 |
| **UAV-M**（多光谱测绘） | 2 | 多光谱 + RGB 补拍相机 | Orin NX 16GB | 正射拼接、植被/水体/道路通行性分析 |
| **UAV-H**（高光谱分析） | 1 | 高光谱成像仪 | Orin NX 16GB | 水质/气体/精细损毁分析 |
| **UAV-C**（通信中继） | 2 | 无特殊传感器 | Orin NX 8GB | 维持 Mesh 网络连通性 |
| **UAV-R**（中继/冗余） | 2 | RGB 相机（备用） | Orin NX 8GB | 备用侦察 + 中继增强 |

> 上表为典型配置示例，实际部署时各类型无人机数量不是固定的——根据灾情规模（受灾面积、被困人员分布密度、水域复杂度）和可用资源动态确定，调度框架需具备对任意异构集群组合的自适应能力。8GB 与 16GB 板卡的异构性是本框架需要处理的关键资源约束。

### 3.3 多模态任务类型体系

#### 3.3.1 任务分类（5 大类、15 种子任务）

```
任务体系
├── A 类：采集任务（Acquisition）
│   ├── A1: RGB 高分辨率航拍采集
│   ├── A2: 热红外扫描采集
│   ├── A3: 重点区域低空补拍采集
│   ├── A4: 多光谱扫描采集
│   └── A5: 高光谱扫描采集
│
├── P 类：预处理任务（Preprocessing）
│   ├── P1: 图像几何校正/拼接
│   ├── P2: 辐射/大气校正（多光谱/高光谱）
│   ├── P3: 影像去噪、配准与拼接
│   └── P4: 热红外温度校准
│
├── I 类：单模态推理任务（Inference）
│   ├── I1: RGB 目标检测（YOLO/RT-DETR，检测淹没区/溃口/道路阻断/被困人员）
│   ├── I2: 热红外人员检测（轻量 CNN，检测被困人员热源）
│   ├── I3: 多光谱损毁评估（NDVI 变化检测 + 语义分割）
│   ├── I4: 二维变化检测（洪前洪后正射影像/遥感底图对比，淹没范围评估）
│   └── I5: 高光谱异常检测（气体泄漏/水质污染）
│
├── F 类：多模态融合任务（Fusion）★ 核心创新
│   ├── F1: RGB + 热红外多模态被困人员定位（CLIP-style 跨模态检索，水域救援）
│   ├── F2: RGB + 多光谱 + 高光谱 二维损毁评估融合
│   ├── F3: 全模态灾情综合研判（多模态 LLM，如 LLaVA/BLIP-2 架构）
│   └── F4: 时空序列融合预测（多时相分析，预测次生灾害风险）
│
└── C 类：通信任务（Communication）
    ├── C1: Mesh 中继维持
    └── C2: 关键数据回传（高优先级压缩传输）
```

### 3.5 核心改进：多任务 DAG 调度与多模态协同

#### 3.5.1 任务流水线 DAG 建模

用有向无环图 $G = (V, E)$ 建模任务依赖关系，其中：
- $V = \{v_1, v_2, ..., v_n\}$ 为任务节点，每个节点包含 $(type, modality, resource\_vector, priority, deadline)$
- $E = \{(v_i, v_j)\}$ 为数据依赖边，表示 $v_j$ 需要 $v_i$ 的输出数据

**典型应急 DAG 示例**（淹没城区区域——任务链 β，共 13 个子任务）：

```
时隙 →
t1:  [A1: RGB航拍采集@UAV-V1]    [A2: 热红外扫描@UAV-V2]    [A4: 多光谱扫描@UAV-M1]
           │                            │                         │
t2:       ├── [P1: 几何校正@UAV-V1]     ├── [P4: 温度校准@UAV-V2] ├── [P2: 辐射校正@UAV-M1]
           │                            │                         │
t3:       ├── [I1: 溃口/淹没区检测@边缘]  ├── [I2: 被困人员热源检测@UAV-V2]  ├── [I3: NDVI变化检测@UAV-M1]
           │                            │                         │
t4:       │    ┌───────────────────────┘                         │
           │    │                                                 │
t5:       ├── [F1: RGB+热红外融合定位@指挥车]                      │
           │    │                                                 │
t6:       │    ├── [F4: 时空序列溃口扩张预测@指挥车]                │
           │    │                                                 │
t7:   [C2: 关键影像回传]──[F3: 全模态洪涝灾情研判@指挥车]           │
           │                        │                             │
t8:       └────────────────────────┤                             │
                                   │                             │
t9:                           [C2: 研判报告压缩回传至后方指挥部]
                                   │
t10:                  [F2: RGB+多光谱+高光谱二维损毁评估融合@指挥车]
                                   │
t11:                  [I4: 洪前洪后变化检测@指挥车]
                                   │
t12:                  [F4: 次生灾害风险预测@指挥车]
                                   │
t13:                  [C2: 最终评估报告回传]
```

> 上述 DAG 共 13 个节点（A 类采集 3 个 + P 类预处理 3 个 + I 类推理 4 个 + F 类融合 3 个 + C 类通信 3 个 = 实际含通信回传 13 个），关键路径长度约 8 个时序步。任务链的实际长度由指挥车根据区域特征在 5~15 范围内动态生成，部分子任务可并行执行（如 t1 的三路采集），部分存在严格依赖（如融合必须在对应模态推理完成后执行）。

#### 3.5.2 多模态资源六维向量

将 DECCo 的 2 维资源向量（CPU, Memory）扩展为 6 维：

$$\mathbf{R}_u = [R_{cpu}, R_{gpu}, R_{memory}, R_{storage}, R_{bandwidth}, R_{energy}]$$

对应地，每个任务 $v_i$ 的资源需求：

$$\mathbf{E}_i = [E_{cpu}, E_{gpu}, E_{memory}, E_{storage}, E_{bandwidth}, E_{max\_latency}]$$

#### 3.5.3 多目标成本函数扩展

在原 DECCo 的 3 项目标基础上扩展为 5 项：

$$C = \underbrace{\lambda_1 C_{load}}_{负载均衡} + \underbrace{\lambda_2 C_{overload}}_{资源过载惩罚} + \underbrace{\lambda_3 C_{latency}}_{通信延迟} + \underbrace{\lambda_4 C_{deadline}}_{★ 时效违反惩罚} + \underbrace{\lambda_5 C_{priority}}_{★ 优先级加权}$$

**新增项详解**：

**$C_{deadline}$（时效违反惩罚）**：
$$C_{deadline} = \sum_{v_i \in V} w_i \cdot \max(0, T_{complete}(v_i) - T_{deadline}(v_i))$$

其中 $w_i$ 与任务优先级正相关，$T_{complete}(v_i)$ 考虑 DAG 关键路径上的累积延迟。

**$C_{priority}$（优先级感知成本）**：
$$C_{priority} = -\sum_{v_i \in V} \frac{priority(v_i)}{T_{response}(v_i) + \epsilon}$$

高优先级任务获得更低的响应延迟时，整体成本降低。

**参数推荐**（灾害应急场景）：$\lambda_1=0.15, \lambda_2=0.25, \lambda_3=0.1, \lambda_4=0.3, \lambda_5=0.2$

> $\lambda_4$ 权重最高——灾害救援中 **时效性意味着生命**。

#### 3.5.4 距离感知与多跳中继网络建模

**（1）节点空间位置模型**

将网络建模为三维空间中的动态拓扑图 $\mathcal{N}_t = (U \cup \{b\}, L_t)$：

- $U = \{u_1, ..., u_K\}$：无人机节点集合，每架 UAV 在时隙 $t$ 具有三维坐标 $\mathbf{p}_i(t) = (x_i, y_i, z_i)$
- $b$：应急通信指挥车（基站），固定位置 $\mathbf{p}_b$
- 无人机位置随时间步更新（按预设航迹或区域扫描路径移动），由此产生时变链路

**（2）距离驱动的链路质量**

对于任意节点对 $(i, j)$，欧氏距离 $d_{ij} = \|\mathbf{p}_i - \mathbf{p}_j\|_2$ 决定链路属性：

| 链路属性 | 计算方式 |
|---------|---------|
| **路径损耗** | $PL_{ij} = PL_0 + 10\alpha\log_{10}(d_{ij}) + X_\sigma$，其中 $X_\sigma \sim \mathcal{N}(0, \sigma^2)$ 为阴影衰落 |
| **接收信号强度** | $RSS_{ij} = P_{tx} - PL_{ij}$ |
| **可达性判定** | 链路存在当且仅当 $RSS_{ij} \geq RSS_{min}$（接收灵敏度阈值） |
| **链路带宽** | $bw_{ij} = f(RSS_{ij})$，离散映射——信号越强带宽越高（如 WiFi 6 MCS 等级） |
| **链路延迟** | $delay_{ij} = d_{ij}/c + queuing\_delay$，$c$ 为光速，排队延迟取决于中继节点当前负载 |

LoS（视距）概率由地形起伏度决定：平坦区域 $P_{LoS} \approx 1$，丘陵/复杂地形中受建筑物和地形遮挡，$P_{LoS}$ 随距离衰减。

**（3）多跳中继路由**

当 UAV 无法直连指挥车时（$RSS_{i,b} < RSS_{min}$），调度器自动寻找经由中继 UAV（UAV-C 或 UAV-R）的多跳路径。

**路由选择算法**：
- 以指挥车 $b$ 为根节点，在可达链路图 $L_t$ 上运行 Dijkstra 最短路径算法（边权重 = $delay_{ij}$）
- 对每架 UAV 计算到指挥车的 $K$ 条最短路径（$K$-shortest paths），供调度器选择
- 多跳路径的**有效带宽**取路径上各段链路带宽的**最小值**（瓶颈链路）：
  $$bw_{path}(u_i \to b) = \min_{(j,k) \in path} bw_{jk}$$
- 多跳路径的**端到端延迟**为各段延迟之和：
  $$delay_{path}(u_i \to b) = \sum_{(j,k) \in path} delay_{jk}$$

**调度决策中的网络感知**：

距离和路径信息使调度器能够在三种执行模式间做量化权衡：

| 执行模式 | 网络条件 | 决策依据 |
|---------|---------|---------|
| **就地边缘推理** | 多跳有效带宽 < 推理数据回传所需带宽 | 避免瓶颈链路阻塞 |
| **传输到指挥车推理** | 直连带宽充足或瓶颈带宽 > 阈值 | 利用指挥车更强算力 |
| **经由中继协作推理** | 部分预处理在源 UAV 完成（如特征提取），压缩后经多跳中继传输结果 | 减少中继链路数据量 |

**数据传输代价（含多跳）**：

原 DECCo 的传输代价公式扩展为路径感知版本：
$$C_{transfer}(v_i, u_j) = \frac{DataSize(v_i) \cdot modality\_factor}{bw_{path}(u_j \to b)} + delay_{path}(u_j \to b)$$

其中 $modality\_factor$ 体现不同模态的压缩比与传输容忍度：
- RGB 图像：可高度压缩（JPEG, factor=0.3），可容忍少量延迟
- 热红外：带宽小，可直传
- 低空正射影像块：数据量较大（factor=0.2 压缩后），优先边缘拼接/筛选后回传
- 高光谱：数据量极大（factor=0.05），**必须边缘预处理**再传输特征

**（4）GNN 拓扑编码输入**

上述距离、链路质量、多跳路径瓶颈带宽作为 GCN 的边特征输入，使调度策略能够学习"距离-带宽-路由"的联合表征：
- 节点特征：UAV 位置 $\mathbf{p}_i$、距指挥车距离 $d_{i,b}$、板卡资源 $\mathbf{R}_i$
- 边特征（直连）：$[d_{ij}, bw_{ij}, delay_{ij}, RSS_{ij}]$
- 边特征（到指挥车，即使非直连）：$[d_{i,b}, bw_{path}(u_i \to b), hop\_count, delay_{path}]$

### 3.6 调度算法升级：从 mA2C 到 mA3C+MHSA

#### 3.6.1 核心改进

| 维度 | DECCo (mA2C) | CASCADE (mA3C+MHSA) |
|------|-------------|---------------------|
| **Agent 架构** | 单 Agent | 多 Agent（每架 UAV + 指挥车各一个 Agent） |
| **状态编码** | 向量拼接 | GNN 编码网络拓扑 + Multi-Head Self-Attention 编码任务 DAG |
| **动作空间** | 选择 1 架 UAV | 并发选择多架 UAV × 多任务（组合动作空间） |
| **协作机制** | 无显式协作 | 通信注意力机制（Agent 间共享隐状态） |
| **任务依赖** | 忽略 | DAG 图神经网络编码，关键路径感知 |
| **优先级** | 无 | 抢占式优先级队列 + 时效约束软硬结合 |

#### 3.6.2 多模态注意力状态编码器

```
状态编码架构:

输入:
├── 任务 DAG 图 G_task ──→ GAT (图注意力网络) ──→ h_task ∈ R^d
├── 网络拓扑图 G_net  ──→ GCN (图卷积网络)   ──→ h_net  ∈ R^d
├── 资源向量 R_all    ──→ MLP                  ──→ h_res  ∈ R^d
└── 多模态数据流特征 ──→ Cross-Modal Attention ──→ h_modal ∈ R^d

融合:
  h_global = MHSA(Concat([h_task, h_net, h_res, h_modal]))
  → 输出: 全局编码状态，输入 Critic 网络计算 V(s)
```

#### 3.6.3 多 Agent 协同决策流程

```
对于时隙 t:

1. 指挥车 Agent:
   - 接收任务 DAG 更新，计算全局状态 h_global
   - Critic: 评估当前全局状态价值 V(s_t)
   - 广播隐状态给所有 UAV Agent

2. 各 UAV Agent (并行):
   - 接收全局隐状态 + 本地观测 (资源、网络、传感器状态)
   - Actor: 输出本地任务接受概率向量 p_i ∈ [0,1]^{|V_pending|}
   - 仅对自身资源可满足的任务输出非零概率

3. 指挥车 Agent 汇总:
   - 收集所有 UAV 的 p_i
   - 通过 Hungarian 算法 / 贪心匹配 将任务分配给最合适的 UAV
   - 动作掩码: 过滤资源不匹配 / 网络不可达的分配

4. 执行 & 反馈:
   - 分配执行，观测 cost C_t
   - DAG 进度追踪: 当前置任务完成时，解锁后继任务
   - 将 (s_t, a_t, C_t, s_{t+1}) 存入经验回放池

5. 集中训练 + 分散执行 (CTDE):
   - 训练时: Critic 可访问全局信息
   - 执行时: 各 UAV Actor 仅依赖本地观测 + 指挥车广播的压缩隐状态
```

### 3.7 应急场景下的特殊机制设计

#### 3.7.1 三级降级运行模式

灾害场景中指挥车或部分 UAV 可能受损，设计三级降级：

| 模式 | 触发条件 | 调度方式 | 能力 |
|------|---------|---------|------|
| **Mode 1: 全功能** | 指挥车 + 4+ UAV 正常 | mA3C 多 Agent 协同 | 全部任务类型 |
| **Mode 2: 半自治** | 指挥车失效，至少 3 架 UAV | 分布式共识 + 轻量启发式 | 采集 + 边缘推理，减配融合 |
| **Mode 3: 最低生存** | ≤2 架 UAV，网络极不稳定 | 预编程自主巡航 + 本地缓存 | 仅采集，网络恢复后批量回传 |

#### 3.7.2 抢占式救援优先级

定义**硬实时救援任务**：热红外被困人员检测（deadline = 2min）。当此类任务进入队列时：

1. 当前正在执行的普通推理任务（如 RGB 全景拼接）被挂起（checkpoint）
2. 释放 GPU 资源给救援任务
3. 救援任务完成后从 checkpoint 恢复

#### 3.7.3 多模态模型自适应压缩

根据当前可用带宽和任务 deadline 动态选择模型版本：

| 模型 | 完整版（高精度） | 压缩版（低延迟） | 切换条件 |
|------|----------------|-----------------|---------|
| RGB 目标检测 | YOLOv8x (68M 参数) | YOLOv8n (3.2M 参数) | deadline < 3min 或带宽 < 5Mbps |
| 热红外检测 | CSPDarkNet | MobileNet-Thermal | deadline < 1min |
| 多模态融合 | LLaVA-13B | LLaVA-1.6-Mistral-7B-int4 | GPU 可用内存 < 4GB |

### 3.8 实验设计建议

#### 3.8.1 仿真环境与任务生成器

实验全程基于仿真环境，不依赖真实航拍或遥感数据集。仿真环境由以下三部分组成：

**（1）灾害场景生成器**

基于参数化模型生成多样化的洪涝灾害场景，确保调度策略的泛化性：

| 参数 | 取值范围 | 说明 |
|------|---------|------|
| 受灾面积 | 10~100 km² | 区域总面积，影响任务链长度与覆盖密度 |
| 淹没比例 | 20%~80% | 水域覆盖率，影响多光谱/高光谱任务计算量 |
| 建筑物密度 | 低/中/高（3 档） | 影响 RGB 目标检测任务密度与被困人员搜索优先级 |
| 被困人员密度 | 1~50 人/km² | 影响热红外搜索任务 deadline 紧迫度 |
| 通信损毁程度 | 0%~90% | 基站损毁比例，决定 Mesh 自组网路由复杂度 |
| 溃口风险等级 | 0~3（4 档） | 0=无溃口，3=活跃溃口，调度时效性与高光谱优先级 |
| 地形起伏度 | 平坦/丘陵/复杂 | 影响无人机航迹规划与通信视距 |

每个仿真 episode 从上述参数空间中随机采样，生成一个独特的洪涝场景实例。

**参数→任务类型触发规则**：并非所有任务类型在每个场景中都需要——参数条件决定哪些模态任务被激活：

| 任务类型 | 触发条件 | 抑制条件 |
|---------|---------|---------|
| A1 RGB 航拍采集 | 始终触发（基础任务） | — |
| A2 热红外扫描 | 被困人员密度 ≥ 5 人/km² | 被困人员密度 < 5 人/km² |
| A3 重点区域补拍 | 建筑物密度 = 中或高 | 建筑物密度 = 低 |
| A4 多光谱扫描 | 淹没比例 ≥ 30% | 淹没比例 < 30% |
| A5 高光谱扫描 | 溃口风险等级 ≥ 2 或 淹没比例 ≥ 60% | 溃口风险等级 ≤ 1 且 淹没比例 < 60% |
| I1 RGB 目标检测 | 始终触发（随 A1 采集） | — |
| I2 热红外人员检测 | 同 A2（有采集才有推理） | 同 A2 |
| I3 多光谱损毁评估 | 同 A4 | 同 A4 |
| I4 二维变化检测 | 淹没比例 ≥ 40% | 淹没比例 < 40% |
| I5 高光谱异常检测 | 同 A5 | 同 A5 |
| F1 RGB+热红外融合定位 | I1 和 I2 同时存在 | 任一缺失 |
| F2 二维损毁评估融合 | I3 和 I4 同时存在 | 任一缺失 |
| F3 全模态综合研判 | 至少 4 种 I 类任务存在 | I 类任务 < 4 种 |
| F4 时空序列预测 | 溃口风险等级 ≥ 2 | 溃口风险等级 ≤ 1 |
| C1 Mesh 中继维持 | 通信损毁程度 ≥ 30% | 通信损毁程度 < 30% |
| C2 关键数据回传 | 始终触发（至少 1 个） | — |

> 触发规则确保仿真场景的"任务需求"与"灾情参数"逻辑自洽——例如低密度农村淹没区不会凭空出现热红外全城搜索任务，无溃口场景不会触发高光谱异常检测。

**（2）区域划分子模块**

场景生成后，依据空间聚类（如建筑物密度热区、淹没深度梯度、溃口距离）将受灾区域自动划分为 **2~5 个子区域**，指挥车为每个子区域生成定制化的任务链。区域特征向量直接决定该区域任务链的模态组成与子任务参数。

**（3）任务链生成器**

每个子区域按照该区域的特征向量和上述触发规则，自动生成一条包含 **5~15 个子任务** 的任务链。生成规则：

- **采集任务（A 类）**：根据触发规则确定所需的采集模态，每种触发模态生成 1 个采集子任务（1~4 个）。任务参数包括覆盖范围、分辨率要求、数据产出量
- **预处理任务（P 类）**：与采集任务一一对应（仅当对应 A 类存在时生成），计算量取决于源数据大小和预处理算法复杂度
- **推理任务（I 类）**：根据触发规则和已生成的 A/P 类任务确定推理类型（2~5 个）
- **融合任务（F 类）**：仅当同时存在 2 种以上相关模态的推理结果时触发（0~3 个）——简单场景可能无融合任务
- **通信任务（C 类）**：C2 数据回传至少 1 个；C1 中继维持仅在通信损毁程度 ≥ 30% 时生成（合计 1~3 个）

各类型的触发数量由参数条件决定，因此不同场景的任务链在总长度和模态组成上差异显著——简单场景（如低密度农村、无溃口、通信完好）可能仅 5~7 个（以 RGB 采集+检测+回传为主），复杂场景（城区高密度、活跃溃口、通信瘫痪）可达 13~15 个（覆盖全部模态 + 多级融合 + 中继维持）。

每个子任务的资源需求向量 $\mathbf{E}_i$（6 维）和 deadline 由生成器根据任务类型和区域紧迫度参数计算，而非从数据集中读取——这保证了仿真可以生成任意规模和组合的任务序列。

**（4）计算资源与网络仿真**

**计算仿真**：每架无人机模拟一块 Jetson Orin NX 板卡（8GB 或 16GB 显存规格），任务的 GPU 显存占用和推理延迟基于目标模型的计算量（FLOPs）与板卡理论算力（TOPS）按比例估算。

**网络仿真——距离感知与多跳中继**：

网络仿真的核心是**空间距离驱动**，所有链路状态从节点间的三维距离推导：

```
仿真时隙 t 的链路更新流程:

1. 位置更新:
   指挥车: 固定位置 p_b
   UAV:   p_i(t) = p_i(t-1) + v_i · Δt  (沿预设航迹移动)

2. 距离计算:
   d_{ij} = ‖p_i - p_j‖₂  (所有节点对)
   d_{i,b} = ‖p_i - p_b‖₂  (各UAV到指挥车)

3. 链路评估 (每对节点):
   PL = 40 + 28·log10(d_{ij}) + N(0, σ²)   (城郊场景路径损耗模型)
   RSS = 20 dBm - PL                       (WiFi 6 典型发射功率)
   可连通 ⟺ RSS ≥ -82 dBm                  (MCS0 最低灵敏度)
   bw = g(RSS)                             (MCS→带宽查表)
   delay = d_{ij} / 3e8 + queuing_delay

4. 多跳路由计算:
   以指挥车为根，在连通图上运行 Dijkstra 求每架UAV的最短路径
   记录: 跳数、瓶颈带宽、端到端延迟
   K=3 条备选路径供调度器选择
```

**关键仿真参数**：

| 参数 | 值 | 说明 |
|------|-----|------|
| WiFi 6 载频 | 5.8 GHz | 无人机间通信频段 |
| 发射功率 $P_{tx}$ | 20 dBm | Orin NX 挂载 WiFi 模块典型值 |
| 接收灵敏度 $RSS_{min}$ | -82 dBm | MCS0 (最低速率) |
| 路径损耗指数 $\alpha$ | 2.8 | 城郊混合场景 |
| 阴影衰落 $\sigma$ | 6 dB | 受建筑物和地形遮挡 |
| 最大通信距离 | ~1.5 km (LoS) | 由灵敏度和路径损耗公式反推 |
| 通信损毁程度 | 0%~90% | 比例随机剔除基站侧的可用链路，模拟基础设施损毁 |

多跳中继的关键约束：每增加一跳，有效带宽由瓶颈链路决定，端到端延迟累加。调度器在决策时必须权衡——经过 2 跳中继回传 8MB 图像的延迟可能远超在本地 Orin NX 上直接推理的耗时。

#### 3.8.2 评估指标

由于实验全程基于仿真（无真实图像/遥感数据），调度质量的评估聚焦于**任务完成时效**和**计算资源利用效率**两个维度，共 6 项指标：

| 指标 | 符号 | 定义 | 物理意义 |
|------|:----:|------|---------|
| **Makespan** | $T_{total}$ | 从一个区域任务链的第一个子任务开始执行到最后一个子任务执行完毕的总时间 | 衡量调度策略对完整任务链的整体吞吐效率 |
| **Average Task Completion Time (ATCT)** | $\bar{T}_{task}$ | $\frac{1}{|V|}\sum_{v_i \in V} (T_{finish}(v_i) - T_{start}(v_i))$，所有子任务从开始到完成的平均耗时 | 衡量单个任务的平均响应速度 |
| **Per-Type Average Completion Time (PTCT)** | $\bar{T}_{type}$ | 按任务类型（A/P/I/F/C）分组计算平均完成时间 | 揭示调度策略对不同模态任务类型的差异化处理能力 |
| **GPU Utilization Mean** | $\bar{U}_{gpu}$ | 所有板卡 GPU 利用率的时域与卡间均值 | 衡量 GPU 计算资源是否被充分利用 |
| **GPU Utilization Std** | $\sigma_{gpu}$ | 所有板卡 GPU 利用率的时域标准差（先按板卡平均，再跨板卡计算标准差） | 衡量负载均衡——标准差越小，GPU 负载越均衡 |
| **Memory Utilization Mean** | $\bar{U}_{mem}$ | 所有板卡显存利用率（已用/总显存）的时域与卡间均值 | 衡量显存资源的利用效率，反映调度器对大模型任务（如融合推理）的显存感知能力 |

**核心评估逻辑**：

一个好的调度策略应当做到：
1. **Makespan 最小**——尽快完成整个任务链，释放无人机集群投入下一区域
2. **ATCT 最小**——每个子任务从提交到完成的等待时间尽可能短
3. **PTCT 最小且不同类型间差异合理**——高优先级类型（如热红外人员检测）的 PTCT 应显著低于低优先级类型
4. **GPU 利用率均值高且标准差低**——各板卡 GPU 充分且均衡地工作，避免某些板卡过载而其他板卡空闲
5. **显存利用率高**——16GB 板卡的大显存优势得到充分利用（大模型融合任务优先分配到 16GB 板卡）

**指标组合使用**：

| 对比场景 | 关键指标 |
|---------|---------|
| 不同调度策略对比 | Makespan + ATCT + $\bar{U}_{gpu}$ + $\sigma_{gpu}$ |
| 不同任务链长度/复杂度 | PTCT（按类型） |
| 不同集群规模（UAV 数量） | Makespan + $\bar{U}_{gpu}$ |
| 消融实验（去掉某模块） | 全部 6 项指标的变化幅度 |

#### 3.8.3 消融实验设计

每组消融实验在相同的仿真场景集上运行，对比完整 CASCADE 与消融变体的 6 项指标变化：

1. **去掉多模态资源感知**：将 6 维资源向量退化回 CPU+Memory 二维（等效 DECCo 资源模型），预期 GPU/显存利用率显著下降，Makespan 延长（大模型融合任务无法感知 GPU 可用性）
2. **去掉 DAG 依赖建模**：将任务链拍平为独立任务序列，忽略数据依赖关系，预期 Makespan 和 ATCT 均延长（无法利用流水线并行，前置任务完成后后继任务无法立即启动）
3. **去掉优先级与 deadline 机制**：统一 FIFO 调度，无抢占，预期高优先级类型（I2 红外搜索、F1 融合定位）的 PTCT 显著增加
4. **去掉 GNN 拓扑编码**：用全连接 MLP 替代 GNN 编码网络拓扑，预期在网络拓扑动态变化剧烈的场景（高通信损毁度参数）中 $\sigma_{gpu}$ 增大、Makespan 延长（调度器无法感知节点间带宽瓶颈）
5. **8GB/16GB 板卡混合 vs 统一 8GB**：将集群中 16GB 板卡替换为 8GB 板卡，观察大显存优势被剥夺后大模型融合任务（F1、F3）的 PTCT 劣化幅度

## 四、总结与展望

### 4.1 核心改进汇总

| DECCo 局限 | CASCADE 改进 |
|-----------|--------------|
| 任务模型仅考虑 CPU/Memory | 6 维资源向量（CPU/GPU/Mem/Storage/BW/Energy） |
| 单任务顺序调度 | DAG 流水线并发调度 |
| 无优先级/Deadline | 硬实时抢占 + 时效违反惩罚 |
| 固定 LAN 网络，无节点位置概念 | 三维距离感知 + 多跳中继路由 + 瓶颈带宽计算 + GNN 拓扑编码 |
| 单 Agent | 多 Agent 协同（CTDE 范式） |
| 无多模态概念 | 多模态数据流感知 + 跨模态模型调度 |
| 无降级机制 | 三级降级运行模式 |

### 4.2 未来研究方向

1. **与大模型深度集成**：将多模态大模型（GPT-4V、Gemini 2.0）作为调度决策的辅助——"语义级任务理解"，而非仅靠数值资源向量
2. **联邦持续学习**：多架无人机在灾害现场持续收集新数据，联邦微调检测模型，适应灾后特有场景（如水域浑浊度变化、淹没植被特征）
3. **人在回路（Human-in-the-Loop）**：救援指挥官的领域知识通过接口注入调度器，实现混合智能决策
4. **数字孪生预训练**：在灾害发生前，利用目标区域的数字孪生环境预训练调度策略，灾时快速迁移（Sim-to-Real Transfer）
